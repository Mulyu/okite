# thinking

企画・計画・設計を練るとき、1周で出した案は前提の取りこぼしや思い込みを含みやすい。**調査 → まとめ → 推敲** の 3 ステップを **最低 3 周** 繰り返し、各周回で前周回を批判的に見直すことで、抜け・矛盾・より良い選択肢に気付く。

## 発動条件

以下のいずれかに当てはまる依頼で発動する。

- 新機能・新ルール・新コマンドの企画
- ドキュメント構成・情報設計・命名体系の検討
- ワークフロー・プロセス・運用手順の設計
- 複数の選択肢から選ぶ必要のある判断
- 「何かを考えて」「案を出して」「どうしたらいい？」のような発散を求める依頼

次のケースでは使わない。

- 事実質問（「この関数は何をする？」）
- 単一の実装タスク（「この関数にログを足して」）
- 一意に決まる修正（既知のバグ修正、タイプミス修正）

## 手順

### 各周回で踏む 3 ステップ

1. **調査** — 関連ファイル・既存仕様・制約・前提を洗い出す。主観でなく根拠を集める段階
2. **まとめ** — 調査結果を構造化し、論点・選択肢・各案のメリット/デメリット/根拠を明文化する
3. **推敲** — 前周回の出力を批判的に読み直す。以下を問う
   - 見落とした前提・制約はないか
   - 仮定が本当に成立するか（反証を探す）
   - もっと良い選択肢・もっと単純な案はないか
   - 既存の仕組みで代替できないか
   - 双方向依存・循環・矛盾が紛れていないか

### 周回の回し方

- **最低 3 周**。案が収束せず新しい論点が出続けるなら追加周回を回す
- 各周回の終わりに「前周回との差分」を一言メモする（何が変わったか、なぜ変わったか）
- 3 周目が 2 周目と実質同じ内容に収束したら、そこで打ち切ってよい

### 出力の出し方

- **検討過程を先、最終案を最後** に置く。結論がスクロール最下部に埋もれないよう、最後に読む位置に最終案を置く
- 中間の 3 周分は `## 検討過程` 配下の `### 1周目` / `### 2周目` / `### 3周目` として冒頭に並べ、各周回は 3〜5 行の要約に圧縮する
- 最終案では「採用した理由」と「不採用にした代替案」を明記する
- ユーザが最短で結論を知りたいときに末尾だけ読めば済むよう、最終案は自己完結させる（検討過程を読まなくても理解できる粒度で書く）

## 出力テンプレート

```markdown
## 検討過程

### 1周目（発散）
- 調査: <何を調べたか>
- まとめ: <どんな選択肢が出たか>
- 推敲: <何に気付いたか>

### 2周目（深掘り）
- 前周回との差分: <何が変わったか>
- 調査: ...
- まとめ: ...
- 推敲: ...

### 3周目（収束）
- 前周回との差分: ...
- 調査: ...
- まとめ: ...
- 推敲: ...

## 不採用にした代替案

- <案A>: <却下理由>
- <案B>: <却下理由>

## 最終案

<採用する案の要点。根拠と想定される効果を添える>
```

## 他スキルとの関係

- **direction（リポジトリ固有）**: そのリポジトリの設計原則・判定フローを定義したスキルが存在する場合、thinking の各周回の「推敲」で direction の判定フロー（ブレスト → 原則で篩う → 一方向性確認 → 判断記録）を篩として通す
- thinking は汎用、direction はリポジトリ固有、という役割分担

## アンチパターン

- 1 周目の案に固執し、2 周目以降が「表現を言い換えただけ」になる → 必ず前提を疑う問いを 1 つ以上立てる
- 調査をせずに推敲だけで周回を回す → 各周回で必ず新しい情報源（ファイル・既存ドキュメント・先行事例）に当たる
- 中間過程をそのまま長文で見せる → 最終案を埋もれさせる。中間は要約のみ
- 検討過程を最終案より後に置く → 結論が末尾で見えづらくなる。**最終案は必ず文末に置く**
