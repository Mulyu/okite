# observability

エージェントはセッションが切れると過去を失い、「今のコンテキストだけで完結する」前提で動きがちで、同じ失敗を繰り返したり、既存の解決策を再発明したりする。**自分の過去実行を自分で観測する作法** を共通スキルとして定義することで、判断前に過去を引き、根拠ベースで動けるようにする（agent self-observability）。

## 発動条件

以下のいずれかに当てはまる依頼で発動する。

- 「前回どうやった？」「過去 PR を見て」「以前の議論」「履歴を確認」などの過去参照依頼
- 「CI が落ちた」「ビルドエラー」「テストが flaky」のような失敗原因調査
- 「メトリクスを見て」「ログを引いて」「監視データ参照」のような運用データ参照
- 「runbook の実行履歴」「定期実行ジョブの結果」など自動運用ツールの状態参照
- 新規実装の冒頭で「既存解決策があるか」を確認するとき
- progress-log を読んだ後、関連 PR / Issue の文脈を補強したいとき

次のケースでは使わない。

- 過去参照が無関係な完全新規実装（直接 implementation へ）
- 単純なファイル読み（直接 Read で完結）
- 観測ツール自体の設定変更（設定変更系のスキル / ハンドリングが担当）

## 観測対象とアクセス手段

実行環境ごとに利用できるツールセットが異なる。冒頭で **その環境で何が使えるか** を確認してから観測する。

### コードホスティング（PR / Issue / Actions 等）

| 知りたいこと | 一般的な手段 |
|---|---|
| PR 一覧・状態・本文・差分・コメント | ホスティング側 API（環境によっては GitHub MCP ツール / `gh` CLI / GitLab API など）|
| Issue 一覧・本文・コメント | 同上 |
| コミット履歴・任意 ref のファイル内容 | 同上、またはローカルの `git log` / `git show` |
| コード横断検索 | ホスティング側のコード検索 API、またはローカル `grep` / `rg` |
| ワークフロー実行・CI ログ | ホスティング側のワークフロー API |

ローカルブランチの状態は `git log` / `git diff` / `git status` で十分。リモート参照は環境で利用可能な API 経由。

### メトリクス・ログプラットフォーム

- 例: Datadog / New Relic / Grafana / Cloud Logging / 自前のログ基盤
- 多くは REST API + API キーで参照可能
- 重い全期間クエリは課金とレート制限の影響を受けやすいので **期間と絞り込みを先に立てる**
- ダッシュボード URL を進捗ファイルや PR 本文に貼る運用と併用

### 自動運用ツール（runbook / オーケストレータ）

- ハーネスに組み込まれた runbook 実行基盤・ジョブスケジューラの実行履歴
- ローカル状態（実行ログのファイル）、または同期先のオブジェクトストレージ（S3 等）
- ツールごとの CLI（例: `<tool> logs` / `<tool> status` 系）

### 静的解析・検査ツール

- ローカルで再実行して現状把握（lint / 型チェック / セキュリティ検査 / 各種静的解析）
- CI 上の結果はワークフロー実行のログから引く

### 進捗ファイル（progress-log）

- `.claude/progress/<task-slug>.md` を Read
- 「セッション履歴」「未解決の問い」「次の一手」を確認

## 手順

### 1. 観測目的の明確化

- 何を判断したいか / 何を再現したいかを 1 行で書く
- 「過去参照が本当に必要か」を問う。新規実装で関係ないなら observability を呼ばない

### 2. 観測対象とツールの選択

目的別に対象を絞る:

| 目的 | 主な対象 |
|---|---|
| 過去の類似実装を探す | コードホスティングの検索 + PR 一覧 |
| CI 失敗の原因調査 | ワークフロー実行ログ |
| レビューでの議論を遡る | PR コメント・review |
| 監視データを引く | メトリクス・ログプラットフォーム API |
| 自動運用の実行履歴 | runbook / オーケストレータのログ |
| 自分の前セッションの状態 | progress-log + git log |

その上で、現在の実行環境で使える具体ツール（GitHub MCP / `gh` CLI / 各社 API クライアントなど）を選ぶ。

### 3. 観測の実行

- **絞り込みクエリを先に立てる** — 取得量が多いと context を浪費する。日付範囲・著者・キーワードで絞る
- **取得は必要最小限** — フル本文ではなく要約・タイトル一覧から始め、必要なものだけ深掘り
- **複数ソース横断** — 1 ソースで結論を出さず、コードと CI、PR と Issue のように 2 ソース以上で裏取り

### 4. 観測結果の記録

- 何を観測して何を判断したかを進捗ファイル・PR 本文・コミットメッセージに残す
- 「過去 PR #X で同じ問題を Y のように解決済み。今回も同じ方針を踏襲」のような形で **参照リンク付きで** 残す
- 観測結果そのものをコミットしない（古くなる）。**参照と判断だけ** 残す

### 5. 観測結果の判断への反映

- 過去解決策がある場合: それを踏襲するか、なぜ変えるかを明示
- 失敗パターンが見つかった場合: 同じ失敗を踏まない手順に組み替える
- 観測しても分からない場合: 「分からない」を進捗ファイルに記録し、ユーザに確認

## 観測クエリのレシピ

### 直近 N 日の自分の PR を引く

PR 検索構文の多くは相対日付をサポートしないため、実行時に N 日前の日付を計算して埋め込む。

```
search query: author:@me created:><YYYY-MM-DD>
# <YYYY-MM-DD> は実行時に N 日前を埋める。例:
#   date -u -d 'N days ago' +%Y-%m-%d   (GNU date)
#   date -u -v-Nd +%Y-%m-%d             (BSD/macOS date)
```

### 特定キーワードで類似実装を探す

```
search query: <keyword> repo:<owner>/<repo>
```

ホスティング側のコード検索が使えない環境ではローカル `rg`：

```bash
rg -n '<keyword>' --type <lang>
```

### 失敗した CI run を引く（最新 N 件）

実行環境で利用可能なワークフロー API を確認し、`status: failure` などで絞り込む。

### メトリクス・ログ API（直近 1 時間のエラーログの例）

プラットフォームごとに API は異なる。Datadog の例:

```bash
curl -s -H "DD-API-KEY: $DD_API_KEY" -H "DD-APPLICATION-KEY: $DD_APP_KEY" \
  "https://api.datadoghq.com/api/v2/logs/events/search" \
  -d '{"filter":{"query":"status:error","from":"now-1h","to":"now"},"page":{"limit":50}}'
```

他プラットフォームでも同等の「期間 + クエリ + 件数上限」のパラメータを揃える。

### 自動運用ツールの直近実行

利用ツールのドキュメントに従う。実行履歴がオブジェクトストレージに同期されている場合は `aws s3 ls` 等で日時 prefix を辿る。

## 他スキルとの関係

- **progress-log** — 観測結果の記録先として最も自然。長期タスクなら必ず併用
- **implementation** — 実装ステップ 1（計画）の冒頭で observability を呼んで類似実装を探す
- **direction** — 4 軸の「運用設計」に対応する代表スキル
- **evaluator** — Evaluator に渡す材料として観測結果を提供できる

## アンチパターン

- **観測せずに「ゼロから新規実装」** → 既存解決策の再発明。冒頭で必ず類似実装を引く
- **観測結果を context に大量投入** → トークン浪費。要約・参照リンクで足りる
- **1 ソースだけで結論** → そのソースが古い・偏っている可能性。2 ソース以上で裏取り
- **観測結果そのものをコミット** → すぐ古くなる。**参照と判断** だけ残す
- **環境で使えないツールを前提に組む** → 冒頭で利用可能ツールを確認してから観測手段を選ぶ
- **重い全期間クエリ** → 課金とレート制限。期間・絞り込みを先に立てる
- **既存の自動運用パターンを観測せずに新規 runbook を書く** → 既存パターンを取りこぼす
- **「観測しても分からなかった」を記録しない** → 次セッションで同じ観測を繰り返す。**分からなかった事実も残す**
