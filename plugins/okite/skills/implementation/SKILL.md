# implementation

実装は手順を飛ばすと「動くが安全でない」「動くが計画とずれている」「動くがテストがない」結果になりやすい。**計画 → 計画のドキュメント化 → アーキテクチャ設計 → monban.yml 更新 → 実装 → テスト → monban all 検証 → 計画照合 → 一時ドキュメント削除と PR** の 9 ステップを順守し、計画を一時ドキュメントに固定して各ステップで参照・更新する。

## 発動条件

以下のいずれかに当てはまる依頼で発動する。

- 新機能・新コマンド・新スキル・新フック・新ルールの実装
- 既存機能の挙動変更・リファクタリング・バグ修正
- 設定スキーマの拡張、CI/CD 改修、ビルド・テスト基盤の改修
- 「実装して」「対応して」「修正して」「追加して」「更新して」のような実装作業依頼

次のケースでは使わない。

- 事実質問・調査依頼（直接 Read/Grep か thinking スキルで完結）
- 計画フェーズだけの依頼（thinking スキルが担当）
- 一行で済むタイプミス修正・コメント追加

## 手順

各ステップが終わったら、次のステップに進む前に必ず **セルフレビュー** を行う。観点と運用は下記「セルフレビュー」節を参照。問題が見つかればそのステップ内で直し、修正が不能な逸脱なら直前のステップへ戻る。

### 1. 計画

- 発散と収束が必要なら先に thinking スキルを呼んで案を絞る
- 既存コードベース・既存仕様・依存関係・制約を確認する
- 受け入れ基準（成功条件）を箇条書きで明文化する

### 2. 計画のドキュメント化

- 計画を `tmp/plan-<short-slug>.md` に書き出す（リポジトリルート直下の `tmp/`。無ければ作成）
- ファイル名は実装内容を表す英小文字スラッグ。日付プリフィクスは不要
- このドキュメントは作業ブランチに **コミットしない**。push 前にステップ 9 で削除する
- 記載項目:
  - 目的・背景
  - 受け入れ基準（チェックリスト形式）
  - 触るファイル一覧
  - アーキテクチャ概要
  - monban.yml への変更点（必要なら）
  - テスト方針
  - リスク・前提

### 3. 計画に基づいたアーキテクチャの設計

- 計画ドキュメントの「アーキテクチャ概要」を具体化する
- 関数・モジュール・データ構造・インタフェース・依存方向を決める
- 不明点・前提の追加が出たら計画ドキュメントを更新してから先に進む

### 4. アーキテクチャに基づいた monban.yml の更新

- 新たに追加・変更するファイルパス・パターン・依存・設定が monban の検査対象に該当するか確認する
- 必要なら `monban.yml` を更新（exclude / forbidden / pinned / allowed など）
- 変更不要なら計画ドキュメントに「monban.yml 変更不要」と明記する

### 5. 実装

- 計画ドキュメントの「触るファイル一覧」に従って編集する
- スコープ外の変更は混ぜない（無関係なリファクタや別バグ修正は別 PR）
- 進捗に応じて計画ドキュメント側のチェックリストを更新する
- コメントは最小限。デフォルト 0 行で、書くのは **WHY が非自明なときだけ**
  - 識別子で表現できる WHAT は書かない（`const ok = ...` の隣に「ok 判定」と書かない）
  - コードを読めば分かる挙動の言い換えを書かない（`clearTimeout(timer)` の隣に「timer をクリア」と書かない）
  - 「なぜこの分岐が必要か」「なぜこの順序か」「なぜ別実装でなくこちらか」だけが書く価値のある内容
  - 迷ったら書かない。レビューで「なぜ？」と聞かれてから足す方が安全

### 6. テスト

- プロジェクトの慣習に従ってテストを追加・更新する（ユニット・統合・E2E）
- 計画した受け入れ基準が満たされることをテストで担保する
- 既存テストを壊していないことも確認する

### 7. monban all による検証

- リポジトリルートで `npx @mulyu/monban all`（または `monban all`）を実行する
- error は修正してから次へ進む。warn は内容を確認して許容判断する
- monban を導入していないリポジトリでは、プロジェクトの lint / type check / test 一式を代替実行する

### 8. 計画ドキュメントとの照合

- 計画ドキュメントの受け入れ基準・触るファイル一覧・テスト方針・monban.yml 変更点と、実装結果を 1 項目ずつ突き合わせる
- 差分があれば次のいずれかを選ぶ:
  - (a) 実装側の不足を埋める（戻ってステップ 5-7 を回す）
  - (b) 計画が古ければ計画側を更新して合意する（更新理由をドキュメントに残す）
- すべて満たされていることを確認してから次へ進む

### 9. 計画ドキュメントの削除とプルリクエスト

- `tmp/plan-<short-slug>.md` を削除する（`tmp/` が空になれば併せて削除可）
- 変更をコミット（必要なら段階的に分割）し、ブランチを push する
- プルリクエストを作成する。本文には:
  - 目的（一時ドキュメントの「目的・背景」を要約）
  - 主な変更点
  - テスト計画
  - リスク・既知の制約

## セルフレビュー

各ステップ完了時に次の観点で振り返り、1 つでも引っかかればそのステップ内で修正してから次へ進む。レビューは Claude 自身が行う。設計の根幹に関わる・複数ステップを巻き戻す必要があるなど重大な問題が見つかったときだけユーザに報告する。

- **計画との一致** — このステップで増減した成果物が計画ドキュメントの該当項目（受け入れ基準・触るファイル一覧・アーキテクチャ概要・テスト方針）と一致するか。逸脱なら計画か実装どちらかで吸収する
- **スコープ** — 依頼外の変更（無関係なリファクタ、別バグ修正、思いつきの追加機能、不要なファイル新設）が混入していないか
- **過剰実装** — 仮想の将来要件・不要な抽象化・不要なエラーハンドリング・後方互換のためだけの分岐・WHAT を二重に語るコメントを足していないか
- **再利用** — 既存の関数・モジュール・設定・スキル・フックで代替できる箇所を新規実装していないか
- **安全性** — セキュリティ脆弱性（コマンドインジェクション・XSS・SQL インジェクション等）、不可逆操作、テスト未保護の重要変更、秘密情報のコミットが含まれていないか

各ステップで参照する観点はステップの性質に応じて取捨してよい（例: ステップ 2 のドキュメント化では「スコープ」「過剰実装」だけで十分なことが多い）。ただし「計画との一致」だけは全ステップで必ず確認する。

## 一時ドキュメントの扱い

- 場所: `tmp/plan-<short-slug>.md`（リポジトリルート直下の `tmp/`）
- ライフサイクル: ステップ 2 で作成 → ステップ 3-8 で更新 → ステップ 9 で削除
- コミットしない: 作業中も push しない。push 前に必ず削除する
- 中断時: ステップ 8 で計画と差異が出て作業を止める場合、計画ドキュメントは残したまま中断し、再開時に同じファイルを読み込む

## 他スキルとの関係

- **thinking**: ステップ 1 で発散と収束が必要なときに先に呼ぶ。implementation の計画フェーズは「収束済みの案を実装計画に落とす」段階
- **direction（リポジトリ固有）**: 設計判断を伴うステップ 3-4 で direction の判定フローを通す
- **documentation**: ドキュメント変更を含む実装ではステップ 5 で documentation スキルの命名・同期ルールを併用
- **simplify / review / self-review**: 実装後の品質チェックは別スキルが担当。implementation は「計画通りに作る」ところまでが責務

## アンチパターン

- 計画をドキュメント化せず頭の中だけで進める → ステップ 8 の照合ができず、抜け・スコープ拡大が起きる
- 計画ドキュメントをコミットしてしまう → ノイズ。push 前に削除する
- ステップ 4（monban.yml 更新）を実装後に回す → 違反検出が手戻りを生む。実装より前に検査ルールを決める
- ステップ 7（monban all）をスキップする → セキュリティ・サプライチェーン違反を見落とす
- 計画と実装のずれを無視して PR を作る → レビュアーが意図を追えない。差分は計画 or 実装側で必ず吸収する
- ステップを並列に進める → アーキテクチャが固まる前に実装に着手すると後戻りが大きい
- コードの WHAT をコメントで二重に書く → 識別子と乖離してすぐ嘘になる。WHY が非自明なときだけ書く
- セルフレビューを 9 ステップ全部終わってからまとめて行う → 後段で見つけた問題が前段の手戻りを生み、コストが膨らむ。ステップ完了ごとに必ず通す
- セルフレビューで「問題なし」と即断する → 観点 5 つを 1 つずつ頭の中で当てて初めて意味がある。思考停止で OK を出さない
