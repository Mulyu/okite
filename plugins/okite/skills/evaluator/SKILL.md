# evaluator

書き手が自分の出力を自分でレビューすると確証バイアスで抜けが見えない。OpenAI 事例の「Generator-Evaluator 分離」（GAN 由来）を mulyu 規約に落とし込み、**別のサブエージェントに評価役を任せる** ことで自画自賛を防ぐ。書き手（Generator）は自分のコンテキストに浸かっているが、評価者（Evaluator）は与えられた観点だけで読むため、書き手が見落とした前提・矛盾・抜けに気付ける。

## 発動条件

以下のいずれかに当てはまる依頼で発動する。

- 「セルフレビューだと甘い」「別観点でレビューさせて」「クリティクして」「批判的に見て」など自己評価への不信を含む依頼
- アーキテクチャ刷新・大規模実装・本番影響あり・セキュリティ機微・データマイグレーションなど **重大な変更** の PR 作成直前
- 設計案を複数立てて選ぶ前の最終確認（thinking の 3 周回を経た後の独立評価）
- monban の静的ルールでは検出できない論点（命名の妥当性、抽象化の妥当性、責務分担の妥当性）の評価が必要なとき

次のケースでは使わない。

- 事実質問・調査依頼
- 人間レビュアー向けの最終レビュー（review スキルが担当。evaluator は **自分のループ内で消費する** 評価）
- 静的に検出できる違反（monban が担当）
- 軽微な変更（1〜2 ファイル・数十行）の PR — evaluator のオーバーヘッドが過剰

## 仕組み

書き手（メインエージェント = Generator）が、Task / Agent ツールで別エージェント（Evaluator）を起動する。Evaluator は与えられた観点だけで成果物を読み、フィードバックを返す。Generator はそれを受けて自分の出力を改修する。

```
Generator → Evaluator → Generator が改修 → （必要なら）再評価 → 完了
```

## 手順

### 1. 評価対象と評価観点の特定

- **対象**: 何を評価させるか（設計メモ / 実装差分 / PR 本文 / monban.yml 案など）
- **観点**: どの観点で見てほしいか（後述「評価観点テンプレート」を参照）
- 観点を絞らずに「総合的に見て」と投げると Evaluator も総花的になる。**3〜5 観点に絞る**

### 2. Evaluator の起動

Agent ツール（subagent_type 指定）で起動する。プロンプトに次を含める:

- 評価対象のパス（または本文を貼り付け）
- 評価観点（リスト）
- 期待する出力フォーマット（観点ごとに 1〜3 行のフィードバック、最後に総合判定）
- **書き手の意図に同調しないよう明示**: 「Generator の意図に引きずられず、観点だけで読め」

```
Agent({
  description: "Independent review of X",
  subagent_type: "general-purpose",  // or "Plan" / "code-reviewer" など状況に応じて
  prompt: <下記テンプレート参照>
})
```

### 3. フィードバックの吟味

Evaluator の出力を **そのまま反映しない**。次を区別する:

- **採用**: 妥当な指摘で書き手が見落としていたもの
- **却下**: Evaluator が文脈を取り違えている / 既に検討済みの論点
- **保留**: 判断材料が足りないもの。追加調査または再評価へ

却下する場合、その理由を 1 行で記録する（PR 本文・コミットメッセージ・進捗ファイルに残す）。

### 4. 改修と再評価

- 採用フィードバックに沿って Generator が成果物を改修する
- 重大論点（設計の根幹に関わる、複数ファイルにまたがる）が指摘された場合は、再度 Evaluator を起動して **同じ観点で** 評価する
- 軽微な指摘は再評価不要

### 5. 評価ループの記録

- 評価結果・採用判断・改修内容を以下のいずれかに残す:
  - PR 本文の「評価ログ」節
  - コミットメッセージ
  - 進捗ファイル（progress-log）の「セッション履歴」エントリ
- 「Evaluator に X を指摘され、Y のように改修した」が読めれば足りる。**Evaluator の出力全文をコミットしない**（ノイズ）

## 評価観点テンプレート

評価対象別の典型観点。状況に応じて取捨選択する。

### 設計案・アーキテクチャの評価

- **前提の検証** — 隠れ前提・暗黙仮定はないか。それは本当に成立するか
- **代替案の網羅** — もっと単純・もっと既存資産で済む案を取りこぼしていないか
- **責務分担** — モジュール・関数の責務が重複・空白なく分かれているか
- **依存方向** — 循環・逆流していないか
- **拡張性 vs 過剰設計** — 不要な抽象化・将来要件の先取りはないか

### 実装差分・PR の評価

- **スコープ** — 依頼外の変更が混入していないか
- **過剰実装** — 仮想要件・不要な分岐・不要なエラーハンドリング・不要なコメント
- **再利用** — 既存関数・モジュールで代替できる箇所を新規実装していないか
- **安全性** — セキュリティ脆弱性・不可逆操作・テスト未保護の重要変更
- **テスト** — 受け入れ基準を網羅しているか、テストが意味のある検証になっているか

### 命名・ドキュメントの評価

- **legibility** — エージェントが業務ドメインを理解できる粒度か
- **一貫性** — 既存命名規約・ディレクトリ構造と整合しているか
- **検索性** — ユーザが探しに来る語で見つかるか
- **冗長性** — 同じ情報が複数箇所で重複していないか

### スキル（SKILL.md）の評価

- **発動条件の精度** — 拾うべき例 / 拾うべきでない例の両方で誤らないか
- **手順の網羅性** — ステップ抜け・順序ミス・スキップしがちな前提
- **他スキルとの境界** — 既存スキルと責務が重複していないか
- **アンチパターンの実用性** — 実際に起こりうる失敗を捕捉しているか

## Evaluator プロンプトテンプレート

```
あなたは独立した評価者として、Generator が作成した <対象> をレビューする。

【評価対象】
<対象のパス、または本文>

【評価観点（この観点だけで評価せよ。Generator の意図に同調しない）】
1. <観点 1>
2. <観点 2>
3. <観点 3>

【出力フォーマット】
## 観点別フィードバック
### 観点 1
- <指摘 1>（重要度: 高 / 中 / 低）
- <指摘 2>（重要度: 高 / 中 / 低）

### 観点 2
...

## 総合判定
- 結論: 採用可 / 要修正 / 再設計
- 根拠: <1〜3 行>
- 致命的論点: <あれば>
```

## 他スキルとの関係

- **monban の self-review** — monban は **静的ルール** ベース。evaluator は **LLM による意味評価**。守備範囲が違うので併用する
- **review スキル** — review は **人間レビュアーに見せる PR レビュー**（外向き）。evaluator は **自分のループ内で消費する評価**（内向き）
- **simplify スキル** — simplify は再利用・品質・効率の観点で **書き手自身が** チェック。evaluator は **別エージェント** に投げる。simplify で済むなら evaluator を呼ぶ必要はない
- **thinking** — thinking の 3 周回後、独立観点での最終確認に evaluator を併用すると効果的
- **implementation** — 重大な変更の PR 直前（ステップ 9 の前）に evaluator を挟む

## アンチパターン

- **観点を絞らず「総合的に見て」と投げる** → Evaluator も総花的になり、深い指摘が出ない。観点は 3〜5 個に絞る
- **Evaluator の出力をそのまま反映する** → Evaluator も間違える。採用 / 却下 / 保留を必ず仕分け
- **Generator の意図を Evaluator プロンプトに長々と書く** → Evaluator が引きずられて確証バイアスが移るだけ。意図は最小限に
- **軽微な変更で毎回 evaluator を呼ぶ** → オーバーヘッドが過剰。実用的には重大変更だけ
- **Evaluator の出力を全部コミット** → ノイズ。採用判断と改修内容だけ残す
- **同じ Evaluator で何度もループする** → 同じバイアスを共有する。再評価が必要なら観点を変えるか別 subagent_type を使う
- **monban で検出できる違反を Evaluator に投げる** → monban で機械的に潰してから evaluator を呼ぶ
