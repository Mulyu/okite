# improvement

スキル(SKILL.md)は使い続けるうちに必ず「発動条件が広すぎ/狭すぎ」「手順に抜け」「ユーザ要望が反映されていない」が出る。改善は **気付いた瞬間** が最も安く、後回しにすると同じ失敗を繰り返す。**振り返り → 原因特定 → 改修案 → 編集 → 自己テスト → PR** の流れを 1 つのスキルに固め、改修の起点（トリガー）を必ず PR に残すことで「なぜそう変えたか」を未来の改修者に伝える。

## 発動条件

以下のいずれかに当てはまるときに発動する。

### A. スキルを使った直後（能動）

スキルが完了した直後（成功・失敗・中断を問わず）、3 点を 1 行で自問する:

- 手順のどこかに無駄・抜け・曖昧さがなかったか
- 発動条件は妥当だったか（過剰発動 / 発動漏れ）
- 完走できなかった場合、その原因は手順側にあるか

「ある」と判定したときだけ進む。なければ何もしない。

### B. スキルの返答に対するユーザフィードバック（受動）

スキルの返答を受け取った後、ユーザが次のような反応を返した場合:

- 否定: 「違う」「そうじゃない」「期待と違う」
- 修正要求: 「もっと〜してほしい」「次からは〜してくれ」「〜は要らない」
- 発動への指摘: 「なぜ発動した？」「これは発動しなくていい」「逆に発動してほしかった」

ただし、成果物そのものの修正依頼（「この関数のここを直して」）は通常の実装タスクであって improvement ではない。スキルの **挙動・手順・発動条件** への不満かを区別する。

### C. 明示依頼

「X スキルを改善して」「improvement で X 見直して」「このスキル直して」など、直接の依頼。

次のケースでは使わない。

- 事実質問（「この関数は何をする？」）
- 通常の実装タスク（implementation スキルが担当）
- スキル本体ではなく成果物そのものへの小修正依頼（その場で直す）
- 1 回限りの事象で、再発条件が立たないもの（n=1 の改修はしない）

## 命名規約

スキル名は他の okite スキル（`thinking` / `documentation` / `implementation`）と揃え、**英語の普通名詞** を用いる。ローマ字や和語をそのまま英字化した名前は採用しない（matcher での識別を統一するため）。

## 対象スコープ

- `plugins/okite/skills/<name>/SKILL.md` — okite プラグインの共通スキル
- `.claude/skills/<name>/SKILL.md` — 呼び出し元リポジトリのローカルスキル
- 両者に同名スキルが存在する場合はユーザに対象を確認する
- 他プラグインのスキル(`plugins/<other>/skills/`)はそのプラグインのオーナーに委ねる。本スキルでは編集しない

## 手順

### 1. 対象スキルと改善トリガーの特定

- どの SKILL.md を改善するか確定する
- 改善のきっかけを 1〜3 行で言語化する:
  - 何が起きたか（事実）
  - 期待とどう違ったか
  - 再発条件（同じ状況で再発するか）
- 再発条件が立たない（n=1）なら improvement を打ち切る。再発するときだけ進む

### 2. 現状の SKILL.md を読む

- 対象 SKILL.md 全文を Read する
- 発動条件・手順・他スキルとの関係・アンチパターンのどこに該当するかを特定する
- 改修候補が「description（発動条件）」「手順」「アンチパターン」のいずれかを決める

### 3. 改善案の検討

- 軽微な修正（1〜2 行の追加・削除・言い換え）はそのまま 4 へ
- 構造的な変更（手順の追加・削除・並び替え、発動条件の改訂、他スキルとの責務再分担）は **thinking スキル** を呼んで 3 周回す。1 周目の案で確定しない
- 改善案は「変更前 → 変更後」の対比で書き、変更理由を必ず添える

### 4. 編集

- Edit ツールで該当 SKILL.md を更新
- 既存の文体・章立て・トーンに揃える（章タイトルは日本語の名詞形、箇条書きは「〜」「〜する」、語尾は常体）
- 関連する他スキル（thinking / implementation / documentation 等）の「他スキルとの関係」節に責務分担の影響がないか確認し、必要なら同 PR で整合させる

### 5. 発動条件の自己テスト

改訂後の `description` で次の 2 種を頭の中でマッチングする:

- 拾うべき例文を 2〜3 個挙げ、いずれもマッチするか
- 拾うべきでない例文（事実質問・無関係依頼）を 2〜3 個挙げ、いずれもマッチしないか

どちらかで誤る場合は description を更に調整して再テストする。

### 6. ドキュメント連動

- ルートの `README.md` / `README.ja.md` に該当スキルの紹介がある場合、説明文を整合させる（documentation スキルの命名・同期ルールに従い、両言語を同 PR で更新）
- スキル名・1 行説明が変わった場合のみ更新。手順の細部は README には書かない

### 7. コミット・push・PR

- ブランチ名: `claude/improve-<skill-name>-<short-slug>`（既存ブランチで作業中ならそのまま）
- コミットメッセージ: `improvement(<skill-name>): <要約>` 形式。本文に「何を変えたか」「なぜ変えたか」「トリガーとなった事象」を 3 段で書く
- push 後、PR を ready for review で作成する。本文には:
  - **トリガー**: どんなフィードバック・失敗・気付きから始まったか（1〜3 行）
  - **変更点**: SKILL.md のどこをどう変えたか、整合させた他ファイル
  - **変更理由**: なぜこの変更が必要か（トリガーとの対応）
  - **発動条件の自己テスト**: 拾うべき例 / 拾うべきでない例とその判定

## PR テンプレート

```markdown
## トリガー
<どんな事象・フィードバック・気付きから改修したか。1〜3 行>

## 変更点
- <SKILL.md のどこをどう変えたか>
- <他に整合させたファイル>

## 変更理由
<なぜこの変更が必要か。トリガーとの対応関係>

## 発動条件の自己テスト
- 拾うべき例: 「<例文>」→ ○/×
- 拾うべきでない例: 「<例文>」→ ○/×
```

## 他スキルとの関係

- **thinking** — 構造的な改修案を検討する手順 3 で呼ぶ。発動条件の根幹を書き換えるときや手順の並びを変えるときは必ず thinking の 3 周回を通す
- **implementation** — improvement は SKILL.md 1 ファイル中心の改修なので 9 ステップを回すほど重くない。複数スキル・フックにまたがる大規模改修になったら implementation に切り替える
- **documentation** — README のスキル紹介を更新する手順 6 で命名・同期ルールを参照。日英両方同 PR で更新する

## アンチパターン

- n=1 のフィードバックでスキル本体を大改造する → 再発条件が立つかを必ず問う。1 回限りの事象ならアンチパターン節に 1 行追加で済むことが多い
- 発動条件をどんどん絞り込んで誰も使えないスキルにする → 改修ごとに「拾うべき例」も自己テストで担保する
- 手順を増やし続けてスキルが肥大化する → 1 つ追加したら同時に「削れる手順」を 1 つ探す。常に総量を意識する
- improvement 自身の改善ループを improvement で回さない → improvement が誤発動・過剰発動・手順抜けを起こしたら improvement で直す（自己適用）
- 命名規約から外れたスキル名を新設する → ローマ字や和語そのままの名前は他スキル(`thinking` / `documentation` / `implementation`)と並びが揃わず、自己発見できない。英語普通名詞に揃える
- 改修理由をコミット/PR に残さない → 次の改修者が判断根拠を失い、同じ議論を繰り返す。トリガーを必ず残す
- ユーザ確認なしに発動条件の根幹を書き換える → 構造的変更は手順 3 の thinking で代替案も提示してから編集する
- 成果物への不満をスキル側の不備と取り違える → 「スキルの挙動・手順・発動条件」への不満なのか「成果物の中身」への不満なのかを区別する。後者は通常の修正タスク
