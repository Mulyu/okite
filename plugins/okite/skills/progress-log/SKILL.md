# progress-log

エージェントはセッションが切れると記憶を失う。短時間で完結するタスクなら implementation スキルの `tmp/plan-<slug>.md`（push 前に削除）で十分だが、**数日〜数週間にわたる長期タスク** や **フェーズに分けてコンテキストをリセットしながら進めるタスク** では、進捗ファイルをリポジトリ内に残し続けて「次のセッションが冒頭でこれを読む」運用にしないと、毎回ゼロから状況把握をやり直す羽目になる。長期タスク用の進捗ファイル方式（`claude-progress.txt` 等で広く知られるパターン）を本ハーネス向けに定型化したもの。

## 発動条件

以下のいずれかに当てはまる依頼で発動する。

- 単一セッションで完結しない長期タスク（数日以上にわたる実装、複数フェーズに分かれる移行作業など）
- 「セッション横断で」「進捗ファイルで管理」「フェーズに分けて」「コンテキストをリセットしながら」のような明示依頼
- 既存の進捗ファイル（`.claude/progress/*.md`）を読んで再開するセッションの開始時
- フェーズ完了で次フェーズの新セッションへ引き継ぐとき

次のケースでは使わない。

- 1 セッションで完結する実装タスク（implementation スキルが担当。`tmp/plan-<slug>.md` で十分）
- 単発の事実質問・短時間の修正
- 計画フェーズだけの依頼（thinking スキルが担当）

## 進捗ファイルの場所と命名

- **場所**: `.claude/progress/<task-slug>.md`（リポジトリルート直下の `.claude/progress/` 配下）
- **命名**: 英小文字スラッグ。日付プリフィクスは不要
- **コミット**: 進捗ファイルは作業ブランチに **コミットする**（implementation の `tmp/plan-...md` と逆。長期タスクの一次資料として扱う）
- **タスク完了時の扱い**: PR マージ前に削除するか、`.claude/progress/archive/` へ移動する（リポジトリの慣習に従う）

## 進捗ファイルの構成

```markdown
# <タスク名>

## メタ
- ブランチ: claude/<branch-name>
- 開始日: YYYY-MM-DD
- 状態: 進行中 | 中断 | 完了
- 担当セッション: <継続的に追記される作業セッションの識別子>

## 目的・背景
<このタスクが解こうとしている問題>

## 受け入れ基準
- [ ] <チェックリスト>
- [ ] ...

## フェーズ
1. [x] <フェーズ 1 名> — <要約> （完了: YYYY-MM-DD）
2. [ ] <フェーズ 2 名> — <要約>
3. [ ] <フェーズ 3 名> — <要約>

## セッション履歴

### YYYY-MM-DD: <セッション要約>
- 実施: <何をやったか>
- 成果物: <作ったファイル・PR 番号など>
- 課題: <未解決の問い・気付き>
- 次の一手: <次セッションでまず何をやるか>

### YYYY-MM-DD: ...

## 未解決の問い
- <設計判断保留中の論点>
- ...

## 参照
- 関連 PR: #...
- 関連 Issue: #...
- 関連ドキュメント: <パス>
```

## 手順

### 1. セッション開始時：進捗ファイルの読み込み

- `.claude/progress/<task-slug>.md` を Read する（ファイル名が分からなければ `.claude/progress/` をリスト）
- 「メタ」「フェーズ」「セッション履歴の最新エントリ」を確認
- **「次の一手」から再開する** — 「セッション履歴を全部読む」「全部やり直す」のは禁止
- 「未解決の問い」に該当する判断を新セッションで下すなら、判断結果を本セッションで履歴に記録する

### 2. 新規タスクの場合：進捗ファイルの作成

- `.claude/progress/<task-slug>.md` を上記構成で作成
- 「目的・背景」「受け入れ基準」「フェーズ」を埋める
- フェーズ分割は 1 セッション = 1 フェーズが目安。フェーズが大きすぎるなら更に分割する

### 3. 作業中：随時更新

- フェーズ内のチェックリスト消化に応じて進捗ファイルを更新する
- 設計判断が発生したら「未解決の問い」または該当フェーズに記録する
- 大きな方針変更は新しいセッション履歴エントリで明記する

### 4. セッション終了時：引き継ぎを書く

セッションを閉じる前に必ず以下を更新する:

- 「セッション履歴」に当セッションのエントリを追加（実施・成果物・課題・**次の一手** の 4 項目）
- 「フェーズ」のチェックリストを最新化
- 「未解決の問い」を最新化
- 「メタ」の状態を更新（進行中 / 中断 / 完了）

「次の一手」は **次のセッションが何も考えずに着手できる粒度** で書く（例: ×「テストを書く」、○「`tests/foo.test.ts` で X ケースを Y のように書く」）。

### 5. フェーズ完了時：コンテキストリセット

- 1 フェーズを完了したら、次フェーズは **新セッションで開始** する（コンテキスト汚染防止）
- 進捗ファイルがあるので新セッションは冒頭で再読すれば足りる
- フェーズ間で必要なら成果物ファイル（設計メモ、データダンプ）も同フォルダに残す

### 6. タスク完了時：進捗ファイルの整理

- 受け入れ基準すべてに ✓ が入ったら「メタ」の状態を「完了」に
- PR マージ前に進捗ファイルを削除するか、`.claude/progress/archive/` に移動する（リポジトリの慣習に従う）
- 重要な設計判断はコミットメッセージ・PR 本文・ADR に転記してから消す

## implementation スキルとの棲み分け

| 観点 | implementation | progress-log |
|---|---|---|
| 想定スコープ | 1 セッション完結 | 複数セッション・長期 |
| ファイル | `tmp/plan-<slug>.md` | `.claude/progress/<task-slug>.md` |
| コミット | しない（push 前削除） | する（作業の一次資料） |
| 主な記載 | アーキテクチャ・受け入れ基準・触るファイル | 上記＋フェーズ・セッション履歴・次の一手 |
| 終了条件 | PR 作成と同時に削除 | タスク完了で削除 or archive |

長期タスクの中の **個別フェーズ** は implementation を呼んで `tmp/plan-...md` で進める。つまり progress-log が外殻、implementation が内殻という入れ子構造になる。

## 他スキルとの関係

- **implementation** — 個別フェーズの実装は implementation の 9 ステップで進める。progress-log は外殻
- **thinking** — フェーズ分割・受け入れ基準の設計時に呼ぶ
- **direction** — 進捗ファイル運用の例外を入れるか判定するときに呼ぶ
- **observability** — 過去セッションの履歴を辿るときに併用

## アンチパターン

- 進捗ファイルを作らずに長期タスクを始める → セッションが切れた瞬間に状況再構築コストが膨らむ
- 「次の一手」を抽象的に書く（「テストを書く」など）→ 次セッションが結局再分析からやり直す。具体ファイル名・具体ケースで書く
- セッション履歴を消す・上書きする → 過去の判断根拠を失う。**追記のみ**
- 進捗ファイルを push しない → 別環境の新セッションが読めない。コミットする
- 進捗ファイルが肥大化して読みづらくなる → セッション履歴は古いものを `## アーカイブ` 節へ畳む。常時表示は直近 3〜5 エントリで十分
- 全フェーズを 1 セッションで詰め込む → コンテキスト汚染で品質が落ちる。フェーズ完了で必ず新セッションに切り替え
- タスク完了後も進捗ファイルを残し続ける → リポジトリのノイズ。完了したら削除 or archive
