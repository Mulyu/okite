# direction

エージェントハーネスは「モデル + 外側全部（ツール・ルール・コンテキスト・検証ループ・権限・状態管理）」の総体である。スキル・フック・検査ルールを足すたびに **4 軸（コンテキスト / 行動 / フィードバック / 運用）のどこに位置し、他軸へどう染み出すか** を判定しないと、ハーネスは雑多な寄せ集めに化けて運用負荷だけが増える。複数プロジェクトを横断する共通ハーネスを運用している場合は、**ハーネス全体に組み込むべきか、個別プロジェクトで持つべきか** の判定も同じ枠組みで篩う。単一プロジェクトのみで利用している場合は「共通化適合性」節を読み飛ばし、4 軸の位置づけと既存資産での吸収可否だけを通せばよい。

なお「4 軸」はハーネスエンジニアリングの整理として広く用いられる枠組みで、本スキル群が採用する判定軸である。別の整理（例: input / control / verify / persist）を採用する場合は本スキルの 4 軸節をそのフレームに置き換える。

## 発動条件

以下のいずれかに当てはまる依頼で発動する。

- ハーネスに新スキル・新フック・新ルール・新検査設定を足す判断
- 既存スキル・フック・検査ルールの改修判断（特に責務分担の変更）
- 同じ仕組みを共通ハーネスに置くか、各プロジェクトに置くかの線引き
- 「これは共通でよい？」「ハーネス全体に組み込むべき？」「プロジェクト固有？」のような設計判断を伴う依頼
- thinking スキルの推敲フェーズで「ハーネス全体への影響」を評価するとき
- improvement スキルで構造的改修を検討するとき

次のケースでは使わない。

- 事実質問
- 単一プロジェクト固有の実装判断（そのプロジェクトの direction が担当）
- スキル文面の軽微な修正（improvement が担当）

## 4 軸

すべての改修案は次の 4 軸のどこに位置するかを必ず特定する。1 つの仕組みが複数軸に染み出すのは普通で、染み出しの **方向と程度** を明示することが重要。

### コンテキスト設計 — 何を見せるか

- エージェントに与える知識・ドキュメント・設定・外部情報源
- 例: スキル本体、ドメイン知識のリファレンス、エージェント向け指示書（`CLAUDE.md` / `AGENTS.md` 等）、MCP サーバ経由の情報
- 主な対象: スキル配布、共通参照ドキュメント、エージェントへの入力チャネル整備

### 行動設計 — 何をさせ、どこまで許すか

- エージェントが取れる行動の語彙と境界
- 例: 使えるツール、権限境界、承認ポイント、ループ上限、サブエージェント分割、コマンドガードフック
- 主な対象: 共通の禁止・許可ルール、ガード系フック、承認フロー

### フィードバック設計 — 出力をどう評価・修正させるか

- 出力に対する評価・修正ループ
- 例: Linter、型チェック、テスト、CI、静的セキュリティ検査、別エージェントによるレビュー
- 主な対象: 静的検査・テストルール、evaluator スキル、レビュー基盤

### 運用設計 — セッション横断でどう回すか

- セッション・プロジェクト・チームをまたいだ継続性
- 例: 進捗ファイル、状態同期、観測可能性、定期実行、配布の仕組み
- 主な対象: progress-log、observability、共通ハーネスの配布そのもの

## 判定フロー

新スキル・新ルールの追加、または構造改修を検討するとき、以下の順で篩う。

### 1. 軸の特定

- 改修案を 4 軸のどこに位置づけるかを 1 行で書く
- 他軸への染み出しを列挙する（「主：行動設計、副：運用設計」のように）
- どの軸にも位置づけられない場合、本当にハーネスの責務かを疑う

### 2. 共通化適合性

- ハーネスを使う全プロジェクトで同じ仕様が望ましいか（= 共通ハーネス行き）
- プロジェクトごとに観点が異なるべきか（= 各プロジェクトの direction 行き）
- 判断軸:
  - **共通化益**: 横断で揃えると運用が楽になるか
  - **固有性**: プロジェクト固有の文脈・ドメインに依存するか
  - **更新頻度**: 全プロジェクト一斉に更新したい頻度か

### 3. 既存資産での吸収可否

順に問う:

1. **既存スキルの改修で済むか** — thinking / implementation / improvement / documentation / direction / progress-log / evaluator / observability のどれかに節を足せば足りるか
2. **既存の静的検査ルールの追加で済むか** — 機械的に検出できる事柄なら新スキルより検査ルールが優先
3. **フックで済むか** — エージェント実行時に機械的に強制したい事柄ならフック
4. 上記すべてが No のときだけ新スキル新設を検討する

### 4. 4 軸全体の整合性

新スキルを足すとき、4 軸のバランスが偏らないかを確認する。

- 同じ軸に重複スキルがないか
- 抜けている軸に対して該当スキルを未来に足す余地があるか

## アンチパターン

- **system-of-record 違反** — チャット・会議メモ・口伝にしか残らない決定をハーネスに流入させる。決定は必ず PR / Issue / コミットメッセージとしてリポジトリに残す（エージェントから見えない情報は存在しないのと同じ）
- **legibility 軽視** — エージェントが業務ドメインを理解できないリポジトリ構造で改修を進める。命名・ディレクトリ配置・README の整備を後回しにしない
- **4 軸の片寄り** — フィードバック設計だけ・運用設計だけが膨らみ、他軸が空洞のまま新スキルを足す。直近の改修が偏ったら、しばらく逆軸の改修だけに絞る
- **共通化機会の取りこぼし** — 「とりあえず各プロジェクトで実装してから」が常態化し、共通化機会を逃す。3 プロジェクトで似た仕組みが現れたら共通化を検討する
- **逆に過剰な共通化** — プロジェクト固有の判断まで共通ハーネスに上げてしまい、各プロジェクトの direction が機能しなくなる。固有性が高いものは個別プロジェクトへ
- **判定軸を持たない新規追加** — 「便利そう」で新スキルを足し、責務が他スキルと衝突する。必ず 4 軸での位置づけと既存資産での吸収可否を通過させる

## 他スキルとの関係

- **thinking** — 推敲フェーズで「4 軸のどこに位置するか」「他軸への染み出しがないか」を篩として通す
- **improvement** — 構造的改修（手順の追加・削除、責務再分担）の検討で本スキルの判定フローを利用する
- **implementation** — 新スキル・新ルールの実装ステップ自体は implementation の 9 ステップで進める。本スキルは「やるべきか・どう位置づけるか」を決める前段
- **各プロジェクトの direction** — 単一プロジェクト固有の判断はそちらが担当。本スキルは **横断判断** が射程
