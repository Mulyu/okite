# direction

okite は「mulyu 配下リポジトリの中央集権アセット置き場」である。スキル・フック・monban ルールを足すたびに、それが本当に **全リポジトリに配布すべき共通資産か**、それとも **個別リポジトリで持つべきもの** かを判定しないと、okite は雑多な寄せ集めに化けて運用負荷だけが増える。ハーネスエンジニアリングの **4 軸**（コンテキスト / 行動 / フィードバック / 運用）を共通の判定軸に据え、すべての改修案を同じ問いで篩う。

## 発動条件

以下のいずれかに当てはまる依頼で発動する。

- okite に新スキル・新フック・新ルール・新 monban 設定を足す判断
- 既存スキル・フック・monban ルールの改修判断（特に責務分担の変更）
- 同じ仕組みを okite に置くか、各リポジトリ（monban / mihari / 他）に置くかの線引き
- 「これは okite で良い？」「mulyu 横断にしたい」「中央集権でやるべき？」のような中央集権判断を伴う依頼
- thinking スキルの推敲フェーズで「okite 全体への影響」を評価するとき
- improvement スキルで構造的改修を検討するとき

次のケースでは使わない。

- 事実質問（「okite には何がある？」→ 直接 Read / Grep）
- 単一リポジトリ固有の実装判断（そのリポジトリの direction が担当）
- スキル文面の軽微な修正（improvement が担当）

## 4 軸

すべての改修案は次の 4 軸のどこに位置するかを必ず特定する。1 つの仕組みが複数軸に染み出すのは普通で、染み出しの **方向と程度** を明示することが重要。

### コンテキスト設計 — 何を見せるか

- エージェントに与える知識・ドキュメント・設定・MCP 情報
- 例: スキル本体、ドメイン知識のリファレンス、CLAUDE.md / AGENTS.md
- okite で扱う典型: スキル配布、共通参照ドキュメント

### 行動設計 — 何をさせ、どこまで許すか

- エージェントが取れる行動の語彙と境界
- 例: 使えるツール、権限、承認ポイント、サブエージェント分割
- okite で扱う典型: monban.yml 共通ルール、guard-bash 系フック

### フィードバック設計 — 出力をどう評価・修正させるか

- 出力に対する評価・修正ループ
- 例: Linter、テスト、CI、別エージェントによるレビュー
- okite で扱う典型: monban 検査、evaluator スキル

### 運用設計 — セッション横断でどう回すか

- セッション・リポジトリをまたいだ継続性
- 例: 進捗ファイル、状態同期、観測可能性、定期実行
- okite で扱う典型: progress-log、observability、中央集権配布そのもの

## 判定フロー

新スキル・新ルールの追加、または構造改修を検討するとき、以下の順で篩う。

### 1. 軸の特定

- 改修案を 4 軸のどこに位置づけるかを 1 行で書く
- 他軸への染み出しを列挙する（「主：行動設計、副：運用設計」のように）
- どの軸にも位置づけられない場合、本当に okite の責務かを疑う

### 2. 中央集権適合性

- 全リポジトリで同じ仕様が望ましいか（= okite 行き）
- リポジトリごとに観点が異なるべきか（= 各リポジトリの direction 行き）
- 判断軸:
  - **共通化益**: 横断で揃えると運用が楽になるか
  - **固有性**: リポジトリ固有の文脈・ドメインに依存するか
  - **更新頻度**: 全リポジトリ一斉に更新したい頻度か

### 3. 既存資産での吸収可否

順に問う:

1. **既存スキルの改修で済むか** — thinking / implementation / improvement / documentation / direction / progress-log / evaluator / observability のどれかに節を足せば足りるか
2. **monban.yml の追加ルールで済むか** — 静的に検査できる事柄なら新スキルより monban ルールが優先
3. **フックで済むか** — エージェント実行時に機械的に強制したい事柄ならフック
4. 上記すべてが No のときだけ新スキル新設を検討する

### 4. 4 軸全体の整合性

新スキルを足すとき、4 軸のバランスが偏らないかを確認する。

- 同じ軸に重複スキルがないか
- 抜けている軸に対して該当スキルを未来に足す余地があるか

## アンチパターン

- **system-of-record 違反** — Slack・Notion・口伝にしか残らない決定を okite に流入させる。決定は必ず PR / Issue / コミットメッセージとしてリポジトリに残す（OpenAI の "system of record" 思想）
- **legibility 軽視** — エージェントが業務ドメインを理解できないリポジトリ構造で改修を進める。命名・ディレクトリ配置・README の整備を後回しにしない（agent legibility）
- **4 軸の片寄り** — フィードバック設計だけ・運用設計だけが膨らみ、他軸が空洞のまま新スキルを足す。直近の改修が偏ったら、しばらく逆軸の改修だけに絞る
- **中央集権の取りこぼし** — 「とりあえず各リポジトリで実装してから」が常態化し、共通化機会を逃す。3 リポジトリで似た仕組みが現れたら okite 化を検討する
- **逆に過剰な中央集権化** — リポジトリ固有の判断まで okite に上げてしまい、各リポジトリの direction が機能しなくなる。固有性が高いものは個別リポジトリへ
- **判定軸を持たない新規追加** — 「便利そう」で新スキルを足し、責務が他スキルと衝突する。必ず 4 軸での位置づけと既存資産での吸収可否を通過させる

## 他スキルとの関係

- **thinking** — 推敲フェーズで「4 軸のどこに位置するか」「他軸への染み出しがないか」を篩として通す
- **improvement** — 構造的改修（手順の追加・削除、責務再分担）の検討で本スキルの判定フローを利用する
- **implementation** — 新スキル・新ルールの実装ステップ自体は implementation の 9 ステップで進める。本スキルは「やるべきか・どう位置づけるか」を決める前段
- **各リポジトリの direction（monban / mihari）** — 単一リポジトリ固有の判断はそちらが担当。okite の direction は **横断判断** が射程
